## 1. Definition

### Simple Definition
A framework is a **reusable, semi‑complete application** that provides a skeleton. You fill in the missing parts to create your own custom application.

### One‑Line Exam Definition
*“A reusable, semi‑complete application with concrete architecture and code, designed to be specialized for specific needs.”*

---

## 2. Why Do We Need Frameworks?

### Problem Without Frameworks
Every project starts from zero – writing everything yourself, repeating common patterns, wasting time.

### Why Were Frameworks Created?
To **reuse** design and code across many projects. Frameworks capture the knowledge of experienced developers.

### What Happens Without Frameworks?
Slow development, inconsistent designs, more bugs.

---

## 3. Framework vs Library – Key Difference

| Aspect | Library | Framework |
|--------|---------|-----------|
| **Control** | You call the library | Framework calls you (Inversion of Control) |
| **Completeness** | Collection of useful functions | Semi‑complete application skeleton |
| **Flow** | Your code controls flow | Framework controls flow |
| **Example** | Java Math library | Spring MVC, React |

**Analogy:** Library = toolbox (you pick tools). Framework = house blueprint (you fill rooms, but structure is fixed).

---

## 4. Framework vs Architecture

| Aspect | Software Architecture | Software Framework |
|--------|----------------------|---------------------|
| **Abstraction** | Abstract (macro‑architecture) | Concrete implementation |
| **Target** | Whole system | Specific domain / application type |
| **Purpose** | Ensure quality, guiding principles | Reusability, specialization |
| **Example** | Layered architecture | Spring MVC (implements MVC architecture) |

**Relationship:** Frameworks are **concrete implementations** of abstract architectures.

## 6. Key Characteristics of Frameworks

| Characteristic | Meaning |
|----------------|---------|
| **Modularity** | Divided into independent parts. |
| **Reusability** | Same framework used for many apps. |
| **Extensibility** | Easy to add new features (hooks). |
| **Inversion of Control** | Framework calls your code, not the opposite. |
| **Non‑modifiable core** | Core code stays unchanged (Open‑Closed Principle). |

---

## 7. Framework Building Blocks: Template & Hook Methods

| Method Type | Purpose | Variability |
|-------------|---------|-------------|
| **Template Method** | Defines skeleton of algorithm (common part) | Fixed |
| **Hook Method** | Place where subclasses can insert specific behaviour | Variable |

**Analogy:** Template = recipe steps (boil water, brew). Hook = add condiments (sugar or lemon).

---

## 8. How Framework Works (Simple Steps)

1. **Framework provides** skeleton (template methods with hooks).
2. **You extend** framework by implementing hook methods (your custom logic).
3. **Framework calls** your code when needed (Inversion of Control).
4. **Result** – your custom application built on framework.

## 9. Simple Java Framework Example (from slides)

### Framework code (cannot be changed)

```java
// Entry point – fixed
public class Main {
    public static void main(String[] args) {
        Human h = new Human(new Walk());  // client provides hook
        h.doMove();
    }
}

// Hook – abstract class, client must implement
public abstract class Move {
    public abstract void action();
}

// Template – framework logic
public class Human {
    private Move move;
    public Human(Move m) { this.move = m; }
    public void doMove() { this.move.action(); }  // calls hook
}
```

### Client code (customisation)

```java
public class Walk extends Move {
    public void action() {
        System.out.println("Walking...");
    }
}
```

**Explanation:** Framework defines `Move` hook and `Human` template. Client provides `Walk` implementation. Framework calls `action()`.

## 11. Role of Frameworks in Enforcing Architecture (from slides)

Frameworks enforce architecture by:

- **Imposing design structure** – you must follow framework rules.
- **Using Inversion of Control** – framework decides flow.
- **Defining constraints** – what you can/cannot change.
- **Providing ready‑made components** – controllers, repositories, etc.
- **Acting as “architectural guardians”** – ensures consistency and maintainability.

---

## 12. Architectural Constraints & Frameworks

- Frameworks **implement and enforce** architectural constraints.
- Example: Spring MVC forces separation into Controller, Service, Repository – a constraint of layered architecture.
- Constraints = rules that limit design choices (e.g., “no direct database call from Controller”).

## 14. Disadvantages / Issues

| Disadvantage | Why It’s Bad |
|--------------|---------------|
| **Learning curve** | Takes time to learn framework. |
| **Initial investment** | Setup and configuration overhead. |
| **Domain expertise needed** | Framework developers must be experts. |
